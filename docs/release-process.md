# Release Process

This monorepo uses [Changesets](https://github.com/changesets/changesets) for versioning and publishing.

---

## 1. Adding a changeset

When your work is ready, add a changeset file to record the intent:

```bash
pnpm changeset
```

This opens an interactive prompt that asks which packages changed and at what bump level (`major`, `minor`, or `patch`). It writes a Markdown file under `.changeset/`:

```
.changeset/
  some-random-name.md
```

Or write it by hand:

```markdown
---
"@maicolextic/bg-subagents-opencode": minor
"@maicolextic/bg-subagents-core": patch
---

Brief summary of what changed and why.
```

Commit the `.changeset/*.md` file alongside your code changes. The file is consumed once — it is deleted when the Version PR is merged.

---

## 2. Version PR flow

On every push to `main`, the GitHub Actions workflow `release.yml` runs `changesets/action@v1`. If any `.changeset/*.md` files are present, it automatically:

1. Opens (or updates) a pull request titled **"Version Packages"**.
2. The PR bumps all affected `package.json` versions and writes `CHANGELOG.md` entries.
3. **Do not merge this PR yourself** unless you are the maintainer doing a release. Merging is the release trigger.

When the Version PR is merged into `main`, `release.yml` runs again and this time hits the `publish` step:

```bash
pnpm -r publish --access public --provenance --no-git-checks
```

All packages with a new version are published to npm under the `@maicolextic` scope.

---

## 3. What happens on publish

The `release.yml` pipeline runs in order:

1. Install dependencies (`pnpm install --frozen-lockfile`).
2. Build all packages (`pnpm -r build`).
3. Typecheck (`pnpm typecheck`).
4. Test suite — gate on 432 green (`pnpm test`).
5. Generate JSON Schema (`node scripts/schema-publish.mjs` → `docs/schema/policy-v1.json`).
6. `changesets/action@v1` — if changesets present: opens Version PR. If versions already bumped (merge): publishes to npm.

The `id-token: write` permission is required for npm provenance attestation.

---

## 4. JSON Schema publish

`docs/schema/policy-v1.json` is generated by `scripts/schema-publish.mjs` on every release pipeline run. GitHub Pages serves the `docs/` directory, making the schema available at:

```
https://maicololiveras.github.io/bg-subagents/schema/policy-v1.json
```

This URL is baked into `PolicySchema.$schema` and into the root README sample. Do not move it without a redirect.

---

## 5. Protocol version discipline

The `PROTOCOL_VERSION` constant in `@maicolextic/bg-subagents-protocol` is the shared wire contract. Changeset rules:

| Change type | Protocol bump | Consumer packages bump |
|-------------|---------------|----------------------|
| New optional field, reserved semantic activates | `minor` | `minor` |
| Breaking schema change, type removal, field rename | `major` | `minor` (adapter must guard) |
| Bugfix, internal refactor | `patch` | `patch` |

Any MAJOR bump to `protocol` must be accompanied by a MAJOR changeset on `protocol` and at minimum a `minor` changeset on all adapters so consumers know to upgrade.

Starting in v0.2, the Claude Code adapter will need its SHA pinned in the Claude Code marketplace manifest. This step will be automated in the release pipeline at that time. For v0.1 (OpenCode only), this step is a no-op.

## Verify before publishing

```bash
# Confirm changeset is valid and lists expected packages
pnpm changeset status

# Run the full test gate locally
pnpm test && pnpm -r typecheck
```
